OpenClaw is designed to run multiple fully isolated agents within a single Gateway process, which is the preferred setup. This means you have one Gateway managing multiple persistent agents, each operating in its own workspace. It’s important to note that this setup does not involve multiple OpenClaw installations; rather, it’s a single running process that houses multiple agents.

Each agent within the Gateway is allocated its own workspace directory, which includes files like SOUL.md, AGENTS.md, USER.md, MEMORY.md, and any associated skills. Additionally, every agent has its own state directory for authentication profiles and configuration, a session store for chat history and routing state, specific tool permissions, and the ability to use different language models. Despite these individual resources, all agents share the same server process and configuration file, simplifying operations by requiring only one Gateway to be run.

There are several reasons to avoid running multiple OpenClaw Gateway processes. Managing multiple Node.js daemons can complicate monitoring and management. Each process would require separate port bindings and health checks, and you would lose the benefits of a shared skills directory and native cross-agent communication. This would lead to a more complex deployment and update process, increasing operational overhead for your team. The single-gateway approach provides isolation where it matters, such as in memory, sessions, tools, and permissions, while keeping operations straightforward.

A few key principles guide the architecture of OpenClaw. First, you have one Gateway with many agents, allowing for shared processes and isolated workspaces. Each agent can be bound to specific channels, listening for messages based on defined routing rules. The order of these bindings is crucial; the most specific rules should be defined first, with a catch-all default at the end. Each agent operates under the principle of minimal privilege, meaning they only have access to the tools necessary for their tasks. Furthermore, different agents can be assigned different models, using more expensive models for complex reasoning and cheaper ones for high-volume tasks.

Agent communication can occur through various patterns, with the simplest method being sub-agent spawning. In this pattern, a coordinator agent can delegate tasks to lightweight sub-agents. For example, if a user requests a weekly summary, the coordinator can spawn several sub-agents to handle specific tasks like querying transaction volumes or summarizing open tickets. It’s advisable to use cheaper models for these sub-agents and to set timeouts to prevent tasks from running indefinitely. Additionally, it's important to include context in the task prompts since sub-agents do not inherit the full context from their parent agent.

Another method of communication is through shared files. This approach allows agents to write data to shared files, which other agents can then read. This is often the most debuggable communication pattern, as you can inspect the files to see what each agent produced. Direct agent-to-agent communication is also possible, but it should be used cautiously due to known issues that can disrupt the functionality of sub-agent spawning.

When it comes to configuration, all agents are managed through a single configuration file. This file outlines the structure for each agent, including their workspaces, models, and binding rules. Binding rules are essential for directing messages to the correct agents based on specific criteria, such as the communication channel or the user involved.

The directory structure for OpenClaw includes the main configuration file, a shared skills directory, and individual workspaces for each agent. Each workspace contains essential files that define the agent's persona, operational rules, user context, and memory management. It’s crucial to avoid sharing authentication directories across agents to prevent session collisions, and sensitive information should be stored in environment variables rather than workspace files.

The heartbeat system allows agents to perform scheduled tasks at configurable intervals. These tasks can be defined in a dedicated file, allowing agents to check for emails, review calendars, or generate summaries on a regular basis. Setting light context for routine tasks can help control costs by reducing the amount of conversation history needed.

Agents can extend their capabilities through skills, which are stored in their workspace or a shared directory. Common built-in skills include access to Google services and content summarization. Hooks are also available to run automatically during agent sessions for memory retention, debugging, and error tracking.
