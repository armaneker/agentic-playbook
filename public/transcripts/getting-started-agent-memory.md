An AI agent is a system designed to perceive its environment through various inputs, such as messages, API data, and files. It can reason over these inputs using a large language model, act on the environment by utilizing tools, and importantly, remember information across sessions by storing and retrieving knowledge from external memory. This capability of memory is what distinguishes an agent from a simple chatbot. Without memory, an agent loses all context as soon as a session ends.

Next, let's discuss the difference between stateless agents and memory-augmented agents. A stateless agent can perceive, reason, and act, but it lacks persistent memory. This means it processes each request in isolation. For example, if a user asks for restaurant recommendations and later wants to book one, the agent won't remember the previous conversation and will require the user to specify which restaurant to book. This leads to several limitations, such as the inability to handle long-horizon tasks, lack of context across sessions, loss of user preferences, and increased costs since the full context must be included in every request.

In contrast, a memory-augmented agent stores interactions and knowledge in an external database, allowing it to retrieve relevant context before responding. For instance, when a user asks to book a restaurant, the agent can pull information from earlier interactions, making the conversation feel continuous and coherent. This type of agent can complete long-horizon tasks, maintain context across sessions, reduce costs by only retrieving necessary information, and ensure reliable multi-step workflows.

Moving on, the simplest form of agent memory is conversational memory. This involves storing the history of interactions, including timestamped messages from both the user and the assistant, in a database. While this allows for basic continuity, it has its limitations. Context windows are finite, meaning they can't hold all the information about user relationships and preferences. Additionally, not all useful information is captured in chat logs; valuable data from workflows and tool outputs also needs to be remembered. Therefore, agents benefit from having structured and queryable knowledge rather than just a flat log of messages.

Agent memory can be categorized into short-term and long-term types, each with its subtypes. Short-term memory includes working memory, which consists of the context window of the language model and any temporary notes made during a session, and a semantic cache that stores previous responses indexed by similarity. This cache allows the agent to quickly retrieve relevant responses instead of generating new ones, saving both time and resources.

Long-term memory is divided into episodic, semantic, and procedural memory. Episodic memory contains timestamped records of past interactions, allowing the agent to recall previous discussions. Semantic memory encompasses the domain knowledge necessary for the agent to perform its tasks, while procedural memory records how the agent completed tasks in the past, enabling it to reference successful methods for similar future tasks.

The core of agent memory resides in three main components. First, there's the large language model, which holds broad but static parametric memory from its training data. Second, the embedding model captures semantic relationships by generating vector representations of text, aiding in both storage and retrieval. Finally, the database serves as the primary infrastructure for storing, retrieving, and optimizing agent knowledge. It manages all types of memory, performs vector searches for relevant information, and supports metadata filtering and CRUD operations through a memory manager.

If you're familiar with retrieval-augmented generation, agent memory follows a similar pattern. It begins with ingestion, where data is chunked, embedded, and stored with metadata. When the agent requires context, it embeds the user's query and matches it against stored vectors. The retrieved context is then combined with the prompt to ground the language model's response in real data. However, unlike standard RAG, agent memory types are stored as separate collections, and the memory manager abstracts the operations, allowing the agent to access memory through tools rather than a fixed retrieval process.

In summary, an agent without memory functions like a stateless chatbot. Memory is essential for enabling multi-step workflows, maintaining context across sessions, and adapting to user preferences. While conversational memory provides a foundation, agents require structured long-term memory to operate effectively. The three types of memory—episodic, semantic, and procedural—serve distinct purposes, and utilizing all three contributes to a capable agent. The database acts as the core of agent memory, handling the majority of data traffic and determining how well the agent can store, retrieve, and apply knowledge.
