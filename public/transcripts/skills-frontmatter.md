The frontmatter is the section at the top of the SKILL.md file, marked by a series of dashes. This part is crucial because it remains in the agent's context window at all times, which is referred to as Tier 1. The agent uses this information to determine whether to activate the skill.

The first field is the name, which is required and has a maximum length of 64 characters. This serves as the skill's unique identifier and should be formatted in kebab-case, meaning no spaces are allowed. For example, a skill could be named "sales-reporting."

Next, we have the description, which is also required and can be up to 1024 characters long. This is the most critical field, as the agent matches user queries against this text to assess the skill's relevance. If the description is poorly written, the skill may never trigger. A good description should clearly state what the skill does using action verbs, specify what triggers it with phrases users might actually say, define the domain it covers, and outline any prerequisites or environment variables needed.

There is also an optional field called tools, which restricts which tools the skill can utilize. If this field is not included, the agent defaults to its standard tool access.

To illustrate the difference between effective and ineffective descriptions, consider these examples. A vague description like "Handles reporting tasks" is not useful because it lacks specificity. Similarly, stating "A comprehensive reporting module with support for multiple output formats and data sources" lists features but doesn't include any trigger phrases. In contrast, a well-crafted description might say, "Pull sales reports from the reporting API. Daily and monthly volume, commission, take rate, customer count, churn analysis by product line. Responds to 'show me today's numbers,' 'monthly comparison,' 'which segment is growing,' 'churn rate.' Requires REPORTING_API_USER and REPORTING_API_PASS environment variables." This example is specific and includes both trigger phrases and domain terms.

For a complete example, you might see a frontmatter section like this: it starts with the name "deploy-checker," followed by a description that explains it checks deployment status across environments and queries the CI/CD pipeline for the latest releases, rollback history, and health checks. It includes trigger phrases such as "is it deployed," "what version," "release status," and "last deployment," and notes that it requires the DEPLOY_API_TOKEN environment variable. The tools field specifies that it can use the bash_tool.

When writing your description, think of it as creating a search index entry. The agent performs keyword matching rather than semantic understanding of the frontmatter. It’s also important to include any necessary environment variables so that the agent can inform the user if they are missing, rather than failing during execution. Lastly, once you publish a skill, keep the name stable, as other skills or automation may reference it.
