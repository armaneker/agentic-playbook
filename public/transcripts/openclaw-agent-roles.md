In a well-structured multi-agent system, each agent is assigned a specific role that aligns with its purpose, model, and tools. Let's explore some example agent roles that cater to common organizational needs.

The Coordinator Agent serves as the central command hub. Its responsibilities include conducting morning briefs, tracking key performance indicators, summarizing cross-functional updates, and performing strategic analyses. For this role, the recommended model is one that excels in complex reasoning and synthesis across various domains. This agent can receive direct messages from the primary administrator and acts as a catch-all for any unmatched messages. It has access to a range of tools, including reading, writing, executing tasks, web searches, and memory searches. It can also spawn sessions to delegate tasks to sub-agents and query outputs from other agents. The workspace files for this agent include a strategic advisor persona, admin preferences, morning brief protocols, and periodic checklists to monitor dashboards for anomalies. A key behavior of this agent is its ability to spawn sub-agents on more cost-effective models for parallel research tasks.

Next, we have the Engineering Agent, which focuses on code reviews, pull request checks, documentation, and technical troubleshooting. This agent is best suited for a model that is fast and efficient in handling code-related tasks. It operates within designated channels for engineering and DevOps, and it receives notifications through GitHub webhooks. The tools it can access include reading, writing, executing code, and processing Git commands, while it is restricted from spawning sessions or controlling gateways. This agent runs in a Docker sandbox to ensure it executes code in isolation. Its workspace files consist of a senior developer persona, PR review standards, and references to repositories and CI pipelines.

Moving on, the Product Agent is responsible for writing specifications, user stories, and documentation, as well as conducting competitive analyses. This agent also uses the same model as the Engineering Agent. It operates in dedicated product and general channels and has access to tools for reading, writing, and searching. However, it does not execute code. Its workspace files include a product manager persona, templates for feature specifications, and documentation platform API access.

The Risk and Analytics Agent focuses on data analysis, financial modeling, and regulatory reporting. It requires a model capable of strong reasoning for analytical tasks. This agent operates in analytics or risk channels and only accepts direct messages from approved analytics team members. It has read access to databases but is restricted from writing to production. The workspace files for this agent include an analyst persona with regulatory knowledge and assessment protocols.

The Sales Agent, which is introduced in Phase 2, handles sales inquiries, generates client reports, and tracks customer relationships. It operates in a dedicated sales channel and uses the same model as the Engineering and Product Agents. This agent has access to reading, writing, and searching tools but does not execute code. It's advisable to deploy this agent after stabilizing the first few agents to minimize coordination complexity.

The Personal Assistant Agent is often the first agent deployed. It manages email triage, calendar scheduling, and daily digests. This agent requires a model that understands nuanced communications and operates through direct messages from the primary user and mobile messaging platforms. It has access to tools for reading and writing, along with integrations for email and calendar management. The workspace files include an executive assistant persona, a morning digest schedule, and user preferences.

The Support Agent focuses on ticket triage, customer question routing, and knowledge base lookups. It operates in a support channel and integrates with platforms like Intercom or Zendesk. This agent uses a conversational model and has access to tools for reading, writing, and searching, but it does not execute code. Its workspace files include a support specialist persona and escalation rules for issues.

Lastly, the Content and Marketing Agent is responsible for drafting social media content, scheduling posts, and monitoring competitors. It operates in a marketing channel and has access to tools for reading, writing, and searching. This agent integrates with social media APIs and content management systems. Its workspace files include brand voice guidelines and content calendar rules.

When selecting models for these agents, the Coordinator and Risk/Analytics roles benefit from a model focused on complex reasoning, while Engineering, Product, Sales, and Support roles are better served by a model that is fast and cost-effective. Sub-agents spawned from the primary agents should also utilize the more economical model to maintain cost control while allowing the parent agent to handle synthesis.
