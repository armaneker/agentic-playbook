The Research-Plan-Implement loop is a structured workflow designed to utilize AI agents as engineering collaborators rather than mere autocomplete tools. 

To start, it's essential to understand the mental model behind this approach. Picture your AI agent as a fast, well-read junior developer who often lacks judgment. This developer can work tirelessly and is willing to rewrite code multiple times without complaint. They've been exposed to countless codebases across various languages and frameworks. However, they don’t possess the business context or the reasoning behind specific architectural decisions you've made in the past. They can produce technically correct code that may not fit your specific needs. The key difference between simply using AI and effectively working with it lies in knowing what tasks to delegate to the agent and what to handle yourself. The RPI loop provides that necessary structure.

Before diving into the loop, it's crucial to manage what the agent sees, which is referred to as context engineering. Every token in the context window incurs a cost, and quality diminishes significantly once the context fills beyond about fifty percent, entering what is termed the "dumb zone." Poor context can corrupt the output more than having no context at all. To manage context effectively, consider four strategies. First, persist information outside the context window by storing it in files or documents, allowing agents to pull in relevant data when needed. Second, be selective about what you bring into the current session; only include what's pertinent to the task at hand. Third, summarize and compress information after lengthy sessions to create a concise overview for the next phase. Lastly, isolate tasks by splitting work across different sessions or agents to prevent unrelated context from accumulating.

If you find yourself going down the wrong path with an agent, be aware that trying to steer it back can lead to context poisoning. Old, incorrect decisions may linger and affect future outputs. In such cases, it's best to start a fresh session rather than attempting to correct a poisoned context.

The first phase of the loop is research, where the goal is to complete system discovery without writing any code. In this phase, you should use a restricted mode that allows the agent to read files and search the codebase but not make edits. This helps avoid premature assumptions that could lead to flawed implementations. During research, focus on system mapping to identify relevant files, APIs, and data structures. You should also engage in pattern discovery to uncover existing conventions in the codebase, brainstorm edge cases to surface unknowns, and trace data flow to understand how changes will impact the system. The output of this phase should be a research document summarizing the current state and proposed technical path, which should be committed to the repository for your own review before proceeding.

Next is the planning phase, which is where you can leverage your engineering time most effectively. The agent will handle execution, but your role is to apply your judgment. A solid plan will include clear scope boundaries to define what the agent should not touch, a detailed list of filesystem changes, verification commands to confirm success at each step, and markers for out-of-scope items. Store this plan in the repository to create an audit trail and provide the agent with a concrete reference during implementation. A well-defined plan allows you to use a smaller or cheaper model for the implementation since the critical thinking has already been done. If your plan is vague enough that the agent needs to make architectural decisions during implementation, it indicates that you need to refine it further.

The third phase is implementation. Start with a fresh session, avoiding any carryover from the research transcript. Provide the agent only with the plan file and the specific files it needs to modify, keeping context low and focused. After each step in the plan, conduct a review using local git diff as a first-pass check before creating a pull request. Run the verification command and review the changes as you would with a junior engineer's pull request. If everything looks good, commit the changes; if not, address any issues before moving on. It's important to recognize when a debugging session becomes lengthy and circular. In such cases, ask the agent to summarize its understanding and progress, then start a new session with that summary. If it feels like things are off track, trust your instincts and initiate a new session.

Configuring the agent's behavior involves three layers. The first layer is modes, which are role-based constraints on what the agent can do. For instance, in ask or research mode, the agent can only read files without making edits. In architect or plan mode, it can outline changes but not implement them, while in code or implement mode, it has full access to edit and run tests. Using the appropriate mode for each phase is critical; avoid granting write access during research.

The second layer consists of rules, which are always-on context that loads with every session. Keep this minimal, including conventions like linting rules and architectural preferences, as well as environment details such as build commands and approval gates that dictate what the agent can do autonomously versus what requires human confirmation.

The third layer involves skills, which are on-demand playbooks for recurring workflows. These are loaded only when triggered, preventing unnecessary bloat in the context window. Good candidates for skills include tasks like changelog updates or API documentation generation.

Additionally, consider auto-approval levels to determine what the agent can do without explicit permission. Some actions, like reading files or running tests, are usually safe for auto-approval, while others, such as writing files or running build commands, should be approved on a case-by-case basis. Always require approval for critical actions like committing code or running destructive commands. Start conservatively and gradually loosen restrictions as you build trust in the workflow.

Finally, the Model Context Protocol, or MCP, extends the agent's reach beyond the IDE. It allows interaction with tools like GitHub for pull requests and issues, documentation for up-to-date framework references, and direct querying of database schema definitions. However, be mindful of the token budget; enabling unnecessary MCP servers can waste tokens and confuse the agent. 

Before starting any task with the agent, remember to isolate tasks to one session, conduct research first in read-only mode, commit the plan file before implementation, monitor context quality, review diffs locally before creating a pull request, persist rules in a dedicated file rather than chat history, trim unused MCPs, and review all outputs as you would with a junior engineer's work.
