The references directory is designed to hold large, static content that your skill requires but should not load into the context window immediately. This content is categorized as Tier 3, meaning it only loads when a specific step in the process calls for it.

In this directory, you should include several types of information. First, API documentation is essential, which encompasses endpoint lists, parameter schemas, and response formats. Next, you should add business rules, which detail calculation methodologies, decision trees, and compliance requirements. Additionally, including few-shot examples is beneficial; these are typically five to ten example inputs and outputs that the agent can use for pattern matching. Lookup tables, such as category codes and rate schedules, should also be present, along with prompt templates that consist of reusable fragments for consistent formatting.

The structure of your directory will typically look like this: within your skill folder, you will have a SKILL.md file and a references folder. Inside the references folder, you might find files like api-endpoints.md for the full API surface, calculation-rules.md for business logic, example-outputs.md for few-shot examples, and category-codes.md for lookup tables.

When you want to reference a specific file in your SKILL.md Steps section, you should point directly to that file. For instance, if you need to apply a calculation, you would specify loading rules from the calculation-rules.md file, matching the input to the correct formula, and returning the result formatted to two decimal places. The key point here is that the agent will only read the file when it reaches that particular step. If the user's query does not trigger that step, the file will not load.

It's important to be specific when referencing files. Avoid vague instructions like "check the references folder." Instead, name the exact file needed, as the agent requires a clear path to follow.

On the other hand, there are certain things you should not include in the references directory. For instance, frequently changing data such as daily rates, live API responses, or real-time metrics should be handled by scripts that fetch fresh data at runtime. Additionally, if you have small snippets of information under 200 tokens, you should place those directly in the Steps section rather than in references. Finally, any executable logic must reside in the scripts folder, not in references.

As an example, consider an API endpoint reference. This file would detail the reporting API endpoints, such as the GET method for daily sales aggregates. It would specify required parameters like the date in ISO 8601 format, optional parameters for filtering by product line, and the response structure, which includes total transaction counts, revenue, and unique customer counts. It would also outline potential errors, such as invalid credentials or rate limits. This file could be quite large, possibly exceeding 500 tokens, so loading it only when the agent needs to make an API call helps keep the context window efficient for other interactions.
