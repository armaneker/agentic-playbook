Connecting agents to local Ollama models allows you to route a specific OpenClaw agent without altering the global defaults. This approach is beneficial because running a local model through Ollama can save on API costs, keep sensitive data within your network, and provide better control over latency. However, it’s important to note that not every agent should be switched to a local model, as cloud models often perform better for complex reasoning tasks. Therefore, the ideal strategy is to implement per-agent routing, maintaining your default provider while only switching those agents that will benefit from a local model.

In the 2026.3 version of OpenClaw, there are some changes that affect configuration validation. Previously functioning configurations may now fail. For instance, standard command-line interface patterns that reference agents may not resolve correctly because agents are stored in an array rather than a keyed object. Additionally, tool-calling protocols may clash with distilled local models that do not support the tools field, resulting in errors indicating that tools are not supported. Fortunately, both issues can be resolved through a three-step process.

First, you need to register the Ollama provider. OpenClaw must recognize the provider before any agent can reference it. Each model entry requires both an ID and a name, and the schema will reject any partial definitions. You will need to specify the base URL, an API key, and the models you want to include. The API key is required by the schema but is not actually used by Ollama, so any non-empty string will suffice. If your Ollama host is on a different machine, ensure you start it to listen on all interfaces, as it defaults to localhost, which prevents remote connections.

Next, you will assign the model to a specific agent. Since agents are stored as an array in the configuration, you will need to target them by their index. You can find the index of the agent you want to modify by listing the agents in verbose mode. Once you have identified the index of the growth-agent, you can set the model for that specific slot. It’s important to use the correct provider/model format, as bare model IDs will not resolve.

The third step involves addressing any tool-calling errors. Smaller, distilled models often do not support the tools API. If OpenClaw sends a request that requires tools, Ollama may return an error indicating that tools are not supported, causing the agent to fail silently. You can handle this in two ways: either switch to a model that is capable of handling tools, or create an alias for the model under a name that OpenClaw's protocol can recognize. This alias allows OpenClaw to send requests using the standard protocol without altering the underlying model weights.

To verify that your configuration changes have taken effect, you can check the agent list again and look for the growth-agent. After that, restart the gateway and send a test prompt. If the agent responds correctly, the routing is functioning as intended. If there are issues, you should check the gateway logs for any schema validation failures or tool-call rejections.

When considering performance, keep in mind that the GLM 4.7 Flash model requires a significant amount of VRAM, specifically between 19 and 21 gigabytes. Running it on a card with only 16 gigabytes will either result in a failure to load or cause the model to spill over to the CPU, which can slow down performance. You should monitor the system's performance using tools like nvidia-smi while the agent is running. Additionally, be aware of network latency; local connections typically add a small delay compared to loopback connections. This may not be significant for single requests, but it can accumulate in multi-turn scenarios. To minimize latency, keep both Ollama and the gateway within the same subnet. Also, note that Ollama keeps models loaded for five minutes by default, so the first request after a period of inactivity may incur a delay. If your agent has bursty activity, consider adjusting the keep-alive setting to prevent the model from unloading.

Local models are particularly useful in specific scenarios. They are ideal when an agent processes sensitive data that should not leave your network, when the task is narrow enough that a smaller model suffices, or when the cost per request is more critical than latency or peak quality. For tasks that involve complex reasoning, long-term planning, or where the quality of the model significantly impacts the output, it’s advisable to continue using cloud models.
