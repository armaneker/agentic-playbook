Progressive loading is a crucial architectural choice that enhances the scalability of skills within an agent. Instead of overwhelming the agent's context window with the complete content of every skill, the system employs a three-tier loading strategy. This approach begins with loading metadata, followed by instructions as needed, and finally, any supporting files only when they are required.

The challenge arises when an agent possesses multiple skills. For instance, if an agent has 50 skills, each containing around 3,000 tokens of instructions and 5,000 tokens of reference material, the total would amount to 400,000 tokens just for skill content. Most context windows cannot accommodate this volume, and even if they could, the clarity of information would suffer due to a poor signal-to-noise ratio.

To address this, the solution is structured in three tiers. The first tier involves skill discovery, where the agent retains the YAML frontmatter of every skill in memory. For 50 skills, this amounts to about 5,000 tokens, which is manageable within any context window. At startup, the agent has access to the system prompt, conversation history, and a skill index that includes brief descriptions of each skill. The agent uses these descriptions to determine which skill aligns with the user's query, making the description field critically important.

In the second tier, when a skill is activated, the agent loads the complete markdown content from the corresponding SKILL.md file. This includes steps, guidelines, and output formats. Importantly, only the skill that matches the query is loaded, not all skills, which keeps the context lean.

The third tier comes into play when specific steps require additional resources. For instance, if a step instructs the agent to reference a particular file, that file is loaded only at that moment. Once the step is completed, the reference content can be discarded from the context if necessary.

When comparing token usage, loading everything upfront would require an immense amount of tokens—400,000 for 50 skills, 1.6 million for 200 skills, and 8 million for 1,000 skills. In contrast, using progressive loading for a typical query results in approximately 8,000 tokens, regardless of the total number of skills. The key takeaway here is that context usage correlates with the complexity of the active skill rather than the total number of skills available.

Several design implications arise from this approach. First, the quality of the description is paramount since it is the only information the agent consistently accesses. Therefore, investing time in crafting effective descriptions is essential. Additionally, the body of the SKILL.md file should remain under 5,000 tokens. If it exceeds this limit, it likely indicates that reference material is being mixed in, which should be separated into dedicated reference files. 

Furthermore, reference files need to be self-contained. When the agent loads a file, it should not require additional files to understand it. Each reference document should be independently useful. Lastly, scripts must return structured output. The agent should be able to parse the output efficiently, with formats like JSON or consistent tables being ideal. Unstructured text complicates parsing and can lead to errors.

To illustrate this in practice, consider a user asking about the churn rate for the previous month. The agent first scans all skill descriptions in Tier 1 and identifies that the sales-reporting skill matches the keywords "churn" and "monthly." In Tier 2, the agent then loads the full instructions from the sales-reporting SKILL.md file. As it processes the request, it determines the necessary parameters: product, metric, and period, without needing any additional resources from Tier 3 at this stage. 

Next, the agent executes a script to fetch the report, which returns the data in JSON format. When it reaches the step that requires the churn calculation definition, it loads the appropriate reference file from Tier 3. After applying the formula, the agent formats the output according to the established rules. Throughout this process, the total context used remains around 8,000 tokens, while the agent can access as many skills as necessary.
