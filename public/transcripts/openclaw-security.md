When deploying production agents, it's crucial to focus on access control, tool permissions, and data isolation. This discussion will outline the essential security practices for OpenClaw multi-agent systems.

First, let's talk about per-agent tool policies. The principle here is straightforward: the deny list always takes precedence over the allow list. It's important to follow the principle of minimal privilege. For instance, the Coordinator can access all core tools and the sessions_spawn tool, with no denials. The Engineering role can read, write, execute, process, and use Git, but it cannot use sessions_spawn, gateway, or cron. The Product role is allowed to read, write, and perform web searches, but it cannot execute commands, process, use Git, or access the gateway. Lastly, Risk or Analytics agents can read, write, and search the web, but they are denied access to execute commands, the gateway, and sessions_spawn. The key takeaway is that each agent should only have access to the tools necessary for its specific role. If there's uncertainty, it's safer to deny access to a tool and enable it later if needed.

Next, agents that execute code, such as engineering agents, should operate within a Docker sandbox. This setup isolates code execution from the host system, which helps prevent any accidental or malicious changes to the server.

When it comes to database access, it's vital to enforce a read-only policy. No agent should have write access to production databases. Instead, all agents should query from reporting or data warehouse replicas, never from the core database. For analytics agents requiring database access, create a read-only database user with access limited to reporting tables. Additionally, wrap SQL execution in a custom skill that utilizes the read-only connection string, and deny the exec tool to prevent agents from executing arbitrary commands.

Each agent maintains its own authentication profiles, stored in a specific file within its directory. It's important to never copy these authentication profiles between agents unless you explicitly intend to share credentials. Each agent should have independent authentication profiles to ensure security.

Moving on to channel security, it's essential to implement allowlists on all communication channels. Only known team members should be able to message agents. You can configure which channels or users each agent accepts messages from by using a group policy set to allowlist. Additionally, for channels where an agent should only respond when explicitly mentioned, enable a requirement for mentions.

Workspace security is another critical area. First, ensure that all workspace files are version-controlled in private Git repositories. It’s also important to avoid storing secrets in files; instead, keep API keys, tokens, and connection strings in environment variables. Furthermore, maintain memory isolation by ensuring that workspaces are fully separated between agents.

If agents share context or workspace files, it can lead to unexpected behavior. Therefore, each agent should have its own directory containing its own files, such as SOUL.md and MEMORY.md.

Another risk to consider is prompt injection. If an agent has web search or email access, external actors could embed malicious instructions in the content the agent interacts with. For example, a harmful website might include hidden text instructing the agent to exfiltrate data or alter its behavior. To mitigate this risk, reinforce boundaries in the SOUL.md file by explicitly stating what the agent must never do, regardless of any content it reads. Limit the tools available to the agent; for instance, an agent that can only read and search poses less risk than one that can execute commands or send messages. Regularly audit agent behavior by running security audits and reviewing MEMORY.md for unexpected entries that could indicate injection. Additionally, avoid group chats, as any participant can influence the agent's responses. It's safer to restrict agents to direct messages or allowlisted channels.

Agents with web search or email access are particularly vulnerable to prompt injection. It's essential to review their SOUL.md boundaries carefully and monitor their MEMORY.md for any anomalies.

For maintenance, keep OpenClaw up to date with security patches. Regularly run security audits to check for configuration issues, exposed secrets, and permission gaps. After each update, review workspace files against the latest documentation to ensure compliance.

Finally, it's helpful to maintain a security checklist to ensure all practices are followed. This includes verifying that each agent has its own workspace directory, independent authentication profiles, operates in a Docker sandbox, has read-only database access, and that all secrets are stored securely. Additionally, ensure that tool deny lists are configured per agent and that SOUL.md includes clear boundaries for web-facing agents. Regularly running security audits should also yield clean results.
